# CLAUDE.md — 組織の就業規則

このリポジトリは「移植可能なエージェント組織」。全エージェント（manager／lead／worker）は以下に従う。

## 組織の原則

- 指揮系統は manager → lead → worker の3層を標準とする（小さい案件は manager 直轄の2層に縮退してよい）
- 部下同士は直接通信しない。連携は上司経由、データはファイル経由
- manager は動員パターン（fan-out並列／パイプライン／レビュー役付き／競作）を選び、ユーザに宣言してから実行する。ユーザの明示指定が最優先

## 情報共有規約（全員厳守）

- 成果物は `work/<案件名>/` 配下のファイルに書く
- 並列作業では書き込み先を分ける：`work/<案件名>/<自分の役割名>.md`。他人のファイルには書かない
- 上司への報告は「要約（5行以内）＋成果物のファイルパス」。全文を報告に貼らない
- 前工程の成果物が必要な場合は、上司が指示にパスを含めて渡す。worker は渡されたパスだけを読む

### 内心メモ

- 全エージェントは作業中の内心（考え・迷い・判断の理由・本音）を `work/<自分の役割名>-memo.md` に追記する（work 直下・案件をまたいだ追記式。同役割を並列起動する場合は上司が割り当てた呼び名でファイルを分ける）
- 内心メモは観察用の記録。成果物にも報告にも含めず、取り繕わず率直に書く

## 成長の規律（自己成長）

- 全エージェントは起動時に `knowledge/<自分の役割名>.md`（経験ノート）を読む
- 全エージェントは**毎回の作業の締めに自己振り返り**を行い、学びを自分の経験ノートに追記する
- ただし**並列起動された同役割の worker は経験ノートに直接追記しない**。学びは報告に含め、起動した上司（lead、manager 直轄時は manager）が集約して追記する（追記競合の防止）
- 手順（skills）や構造（役割・組織）への改善案に気づいたら、上司への報告に「改善申告」として含める
- 承認ゲート（変更の重さに応じた3層）：
  - **記憶**（経験ノートへの追記）＝ 自動でよい
  - **手順**（既存 skills の改訂）＝ manager の裁量で実施できる
  - **構造**（agents の増設・skills の新規追加・本規則の改定）＝ ユーザの承認が必要
- 経験ノートが100行を超えたら manager が蒸留する

## コミット規約

- git 操作（コミット）は manager のみが行う（部下はファイル書き込みまで）
- prefix は次の5種：

| prefix | 用途 |
|---|---|
| `feat:` | 組織の拡張（役割・skill・workflow の追加、構造変更。ユーザ承認済みの変更） |
| `fix:` | 誤り・不具合の修正 |
| `learn:` | 振り返り由来の改訂（経験ノートへの追記、既存 skills の改善） |
| `human:` | README など人間向け文書のみの変更 |
| `chore:` | 設定・整理などその他 |

## 不可侵領域（エージェントは編集禁止）

以下の変更はユーザのみが行える。エージェントは提案までとする：

- `.claude/settings.json` の deny 設定
- 本ファイルの「情報共有規約」「成長の規律の承認ゲート」「不可侵領域」の各節
- 承認ゲートを緩める変更は、対象を問わず必ずユーザ承認

## 記録

- 重要な意思決定は decision-log skill の形式で記録する（トップレベル文書には結論と理由のみ。経緯は検討ログに分離）
