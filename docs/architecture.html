agent-org / architecture

組織と、組織を動かす仕組み

移植可能なエージェント組織の設計サマリ。リポジトリ1つが「組織」であり、cloneと起動コマンド1行でどの環境でも同じ組織が立ち上がる。

1. 組織のかたち（標準3層）

上司 manager がユーザの窓口。中間管理職 lead が実働の worker を束ねる。同じ役割定義から複数人を並列起動できる。

  • 部下同士は直接通信しない。連携は上司経由、データはファイル経由
  • 部下を持てるかは役割定義の tools（Agent の有無）で構造的に制御する

2. 組織の解剖 — ファイルが制度そのもの

組織の実体はすべてプレーンファイル。だから git で持ち運べ、編集すれば組織が変わる。

制度（メタファー）実体役割
就業規則CLAUDE.md行動規範・情報共有規約・成長の規律・不可侵領域
職務記述書.claude/agents/*.mdmanager／lead／worker の役割・使えるツール
業務マニュアル.claude/skills/動員パターン4種＋意思決定の記録手順
定型業務.claude/workflows/繰り返す仕事の固定手順（運用で追加）
権限規程.claude/settings.jsonpermissions（allow／deny）
経験ノートknowledge/<役割>.md振り返りで得た学びの蓄積（git管理＝学習成果ごと移植）
取引先リスト.mcp.json ＋ .env外部接続の定義。認証だけ環境変数で注入
作業場work/案件ごとの成果物置き場（git管理外・書き捨て）
内心メモwork/<役割>-memo.md作業中の思考・本音の自己申告（観察用。成果物・報告には含めない）

3. 仕事の流れ（タスク駆動）

ユーザはお題を渡すだけ。パターン選択は manager の職務で、選択と理由を宣言してから動く（明示指定があればそれが最優先）。

1 依頼ユーザがお題を渡す
→
2 宣言manager が動員パターンと理由を宣言
→
3 実行部下を並列起動。成果物は work/ へ
→
4 報告要約5行＋ファイルパスで上へ
→
5 集約manager が検品し、結論ファーストで報告
  • 並列作業は書き込み先を worker 毎に分離（衝突防止）
  • 前工程の成果が要るときは、上司が指示にパスを含めて渡す（又聞き禁止）

4. 動員パターン（4種のメニュー）

fan-out 並列

同質の作業が大量にあるとき（一括レビュー・網羅調査）
上司 ─┬→ worker ×N（分担して同時に）

パイプライン

前工程の成果が次の入力になるとき（調査→設計→実装）
01-調査 → 02-設計 → 03-実装（パスで受け渡し）

レビュー役付き

品質担保が必要な成果物（納品物・対外文書）
作業者 → draft → 検査役 →（差し戻し≦2回）

競作

正解が一つでない仕事（企画・設計・命名）
案A / 案B / 案C → 評価基準で比較 → 採用

迷ったらコストの低い順（fan-out → パイプライン → レビュー役付き → 競作）に検討する。定型化した仕事は workflows に焼き込み、選択自体を不要にする。

5. 成長の仕組み（自己成長ループ）

振り返りは行事ではなく全エージェントの作業単位の職務。学びはファイルに書かれ、コミットとして歴史に残る。

層変わるもの承認コミット
記憶knowledge/（経験ノート）への追記自動でよいlearn:
手順既存 skills の改訂manager の裁量learn:
構造agents 増設・skills 新規追加・就業規則の改定ユーザ承認feat:
  • 不可侵領域：settings.json の deny／情報共有規約／承認ゲート規定は、エージェント編集禁止（変更はユーザのみ）。自己編集のドリフトに対する歯止め
  • 肥大化対策：経験ノートが100行を超えたら manager が蒸留する
  • 巻き戻し：git 履歴が成長の記録。悪化した進化は revert で戻せる

6. 記録の規約

prefix用途
feat:組織の拡張（役割・skill・workflow の追加、構造変更。ユーザ承認済み）
fix:誤り・不具合の修正
learn:振り返り由来の改訂（経験ノート追記・既存 skills の改善）
human:README など人間向け文書のみの変更
chore:設定・整理などその他

git 操作は manager 専任（並列コミットの衝突防止）。コミットログを見れば「誰の権限で組織が変わったか」を追える。重要な意思決定は decision-log skill の形式（結論＋理由をトップレベルに、経緯は検討ログに分離）で記録する。

7. 移植の仕組み

git clone <このリポジトリ> && cd agent-org
cp .env.example .env       # 必要な認証情報を設定
claude --agent manager     # 組織が立ち上がる
持ち運ばれるもの移植先で用意するもの
役割定義・就業規則・業務マニュアル・権限規程・経験ノート（学習成果）・git履歴（成長の記録） 認証情報（.env）／個人設定（settings.local.json・git管理外）

8. 関連システム — 観察ゲーム

この組織を舞台に、エージェントの働き・成長を「神の目」で観察するゲームを別途構築予定。組織側は内心メモ規約を実装済み。観察層は組織本体に手を入れず上から被せる（hooks は settings.local.json 側に注入、日報の原料＝成果物と内心メモは git 管理外のためゲーム側でアーカイブ）。
